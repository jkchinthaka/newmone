# RBAC & Platform-Scope Migration

Status: In progress - authorization hardening on branch `fix/enterprise-production-hardening`.

This document is the working inventory for RBAC, platform-scope and object-level
authorization hardening. The authoritative, machine-generated route matrix lives
in [`../go-live/backend-rbac-audit.md`](../go-live/backend-rbac-audit.md) and is
regenerated by `npm run audit:rbac:report`. CI enforces it with `npm run audit:rbac`.

## Baseline (before this migration)

The legacy audit generator (`scripts/generate-backend-rbac-audit.mjs`) reported
24 "TODO" routes. Investigation showed the majority were false positives: the
generator only inspected decorators placed after the `@Get/@Post` decorator, so
`@Public()`, `@Roles(...)` and `@Permissions(...)` written above the HTTP verb
were missed (e.g. `POST /auth/logout`, `POST /auth/invite/accept`, and every
`@Roles`-guarded `/tenants/:id/invitations` route).

A new scanner (`scripts/audit-rbac.mjs`) associates all decorators preceding a
handler (including multi-line decorators) and also scans farm inline controllers
declared in `*.module.ts`. It found 641 routes and only 3 genuinely unscoped
routes:

| Method | Route | Resolution |
|--------|-------|------------|
| POST | `/auth/logout-all` | `@SelfService()` (revokes the caller's own sessions) |
| GET | `/auth/me` | `@SelfService()` (caller's own profile) |
| GET | `/entitlements/me` | `@SelfService()` (caller's own tenant entitlements) |

## Scope model

Every route now carries exactly one explicit authorization scope:

| Scope | Decorator(s) | Meaning |
|-------|--------------|---------|
| public | `@Public()` | Unauthenticated transport; no business data. |
| public-webhook | `@Public()` + `@PublicWebhook(provider)` | Unauthenticated integration webhook; authenticated by provider signature. |
| self-service | `@SelfService()` | Authenticated; acts only on the caller's own resources (subject derived from JWT, never client-supplied). |
| tenant | `@Roles`/`@Permissions` (+ optional `@TenantScoped()`) | Tenant business route; requires active tenant + role/permission. |
| platform | `@PlatformScoped()` + `@Roles('SUPER_ADMIN')` | SUPER_ADMIN platform administration. |

## Platform / cross-tenant surfaces

The `platform-super-admin` and `shared-reference-data` entries in
`scripts/tenant-audit-exceptions.json` cover services that read across tenant
scope for platform reporting (go-live, post-go-live, QA, delivery-readiness,
management-intelligence, reports, pilot-rollout, work-order-taxonomy). Their
controllers are permission-gated (`go_live.*`, `delivery.*`, `REPORT_ROLES`,
`MANAGEMENT_ROLES`, ...) and the `TenantContextGuard` forces a resolvable active
tenant for non-SUPER_ADMIN callers, so the dual-scope query path is not reachable
as an anonymous cross-tenant read.

Remaining work (tracked, not yet done): convert those services to a fail-closed
`requireTenantId()` tenant path plus an explicit `@PlatformScoped()` platform path,
so the tenant-audit exceptions can be removed rather than documented. Until then
the entries are retained with compensating controls (see
`docs/security/platform-scope-policy.md`).

## Object-level authorization

Existing object-level controls remain in force (work-order status transitions,
inventory maker-checker, gate-out override reason+audit, compliance verification,
evidence governance - see `docs/go-live/backend-rbac-audit.md` high-risk
checklist). A full separation-of-duties matrix across every module is tracked as
remaining work in `PRODUCTION_READINESS_REPORT.md`.
